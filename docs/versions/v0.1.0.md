## 2026-04-24（续 — Day 7）

### 完成

**Day 7 — 前端：ChatPanel 流式回答 + [N] 角标**

新增：
- `src/types/chat.ts` —— ChatMessage / ChatRole（user/assistant/error）/ ChatStreamStatus（idle/streaming/done/error）
- `src/hooks/useChatStream.ts` —— 自写 hook，POST `/api/v1/chat/stream`，Stream-decoder 按 `0:` `2:` `d:` `3:` 行级前缀解析 Vercel AI SDK Data Stream Protocol。incremental decode 用 `TextDecoder({stream:true})` + buffer 切行避免 chunk 边界问题；含 AbortController 支持 stop
- `src/features/stock/ChatStream.tsx` —— 对话气泡渲染：
  - user 右对齐 primary 色块
  - assistant 左对齐，文本用 `CITATION_REGEX = /\[(\d+)\]/g` 切，匹配的 `[N]` 用 CitationLabel 组件渲染，未匹配的当文本
  - error 红框
  - 流到一半（content 空）显 spinner

重写：
- `src/components/ChatPanel.tsx` —— 把 Day 5 的占位换成真正的输入 + 流式输出
  - 顶部 header 显当前上下文 ticker
  - `collectCitations(cards)` 从所有 workspace 卡片聚合 citations、按 url 去重、重新打 `[1] [2] [3]` 标签
  - `useEffect(reset, [ticker])` 切股票时清空对话
  - `useEffect(scrollBottom, [messages])` 新消息自动滚到底
  - 流式中 Send 按钮变 Stop 按钮
  - 未选 ticker 时输入框 disabled + placeholder 提示

### 决策

- **不用 @ai-sdk/react 的 useChat**：它默认期望 `{messages: [{role, content}]}` 格式，而我们 backend ChatStreamRequest 是 `{message, cards, citations}`。可以用 `experimental_prepareRequestBody` 转译，但不如自己写 80 行 hook 简单透明。AI SDK 包没白装 —— `ai` package 的协议规范（0:/2:/d: 前缀）我们 backend 已实现，前端 hook 解析的就是它
- **CitationLabel 渲染走 regex 切分而非 markdown parser**：`[N]` 是简单 token，不需要引入 react-markdown（+30KB）。正则 `\[(\d+)\]` 单遍切，未匹配的 `[N]` fallback 当文本，鲁棒
- **collectCitations 在前端重新打编号 [1] [2] [3]**：workspace 三张卡可能各有 citation 用 [1]，聚合后重打编号避免冲突。这是 v0.1 的简化，v0.2 评估改为 backend 统一打编号
- **切 ticker 自动 reset chat**：v0.1 单 workspace 单对话；v0.4 多 workspace 时改为按 workspace 持久化对话历史
- **流式中 Send 变 Stop**：UX 借鉴 ChatGPT；用户随时打断，避免长答非所问

### 遗留

- `useChatStream` 没有 retry / exponential backoff；Anthropic 限流时直接报错，v0.5 加
- ChatPanel 的对话历史只在 component 内部 state，刷新即丢；v0.4 持久化 SQLite 时迁移到 chatStore
- CitationLabel 仍然是 Day 6 的占位 —— 点击直接 `window.open(url)`；Day 8 接 citationStore.open() 切到 drawer
- 没单测；流式 hook 比较难测（fetch + Reader API mock），v0.5 加 vitest + msw 时一起补
- 前端没接收 backend 在 finish chunk 发的 citations data part 后做"按返回的精确编号显示"；当前是用 workspace 卡片聚合 citations 重排作为参照表，依赖 backend 渲染时的编号一致性。短期没问题，但如果 LLM 引用了 workspace 之外的 [N]，前端会渲不出来 → 留 v0.5 重设计

### 决策

- **LangGraph v0.1 只做 prompt 准备，不做 streaming**：streaming 走 Anthropic SDK 直连。理由：langchain-anthropic 的 ChatAnthropic 与我们 salvaged LLMProvider 抽象错位；一个 LLM call 没必要走完整 graph 状态机；先把"准备 → 流"流水线跑稳，v0.2+ 多 agent 时再把 streaming 也搬进 graph
- **采用 Vercel AI SDK Data Stream Protocol** 而非自定义 SSE：前端 useChat 开箱即用，省 Day 7 的协议联调；协议简单（行级前缀 + JSON），Python 这边手写 ~30 行就够
- **Anthropic AsyncAnthropic（不是 sync 包 to_thread）**：FastAPI 全异步栈下原生 async client 更顺，避免线程池调度延迟
- **chat orchestrator 不复用 salvaged llm/factory.py**：streaming 接口不在 LLMProvider 抽象里（complete 是 sync 一次返回）；v0.2 给 LLMProvider 加 stream_complete 抽象方法时再 unify
- **ChatStreamRequest 保留 citations 字段**：前端会带回当前 workspace 的引用源（[N] → URL）；后端用它注入 prompt + 在 finish chunk 回传
