## 2026-04-24（续 — Day 4）

### 完成

**Day 4 — 后端：chat + LLM (LangGraph + Anthropic streaming + Vercel AI SDK 协议)**

Prompts:
- `backend/data/prompts/stock/system_prompt.j2`：FinPilot 助手 system；强制 `[N]` 角标引用，禁编造数字，禁出投资建议
- `backend/data/prompts/stock/follow_up.j2`：把 cards（financial_kpi / announcement_timeline / research_report 三种 schema）+ citations（编号源）+ user_message 注入；trim_blocks/lstrip_blocks 控空白
- `backend/data/prompts/stock/company_overview.j2`：v0.2 占位（让 LLM 给公司主页生成"分析师视角的 1 段话总结"）

Orchestrator:
- `backend/services/chat/orchestrator.py`：
  - LangGraph 单节点 graph（`prepare` 节点）—— v0.1 一个 LLM 调用，graph 主要起 state 管理 + prompt 渲染脚手架；v0.2 加 industry/market agents 时直接 add_node 不重写 service
  - Anthropic AsyncAnthropic 流式 chat（`async with client.messages.stream`）
  - 三种 ChatChunk：delta（文本片段）/ finish（带 citations + finish_reason）/ error
  - 模块级 `_chat_graph` singleton + `_prompt_env` Jinja2 Environment 复用

Route:
- `backend/routes/chat.py`：
  - `POST /api/v1/chat/stream` 接受 ChatStreamRequest（message + cards + citations）
  - 输出 **Vercel AI SDK Data Stream Protocol**：
    - `0:"text"\n` 文本 chunk（JSON-encoded，ensure_ascii=False 中文不转义）
    - `2:[{"citations": [...]}]\n` 数据 part（finish 前发，前端拿来渲抽屉）
    - `d:{"finishReason":"stop","usage":{}}\n` finish part
    - `3:"err"\n` 错误 part
  - Content-Type `text/plain; charset=utf-8` + Header `x-vercel-ai-data-stream: v1`
  - ANTHROPIC_API_KEY 缺失 → 503

测试：12 新增（test_chat_orchestrator ×5 + test_chat_route ×7）→ 58 tests 全绿
- prepare_node 渲染中英 prompt 含/缺 cards 都通
- ChatOrchestrator.stream 用 _FakeAsyncAnthropic 模拟 streaming，验证 delta + finish 流
- 协议编码三种 part 的 byte-level 输出
- 端到端 POST /api/v1/chat/stream 输出协议格式正确
- ANTHROPIC_API_KEY 缺失 → 503

### 决策

- **LangGraph v0.1 只做 prompt 准备，不做 streaming**：streaming 走 Anthropic SDK 直连。理由：langchain-anthropic 的 ChatAnthropic 与我们 salvaged LLMProvider 抽象错位；一个 LLM call 没必要走完整 graph 状态机；先把"准备 → 流"流水线跑稳，v0.2+ 多 agent 时再把 streaming 也搬进 graph
- **采用 Vercel AI SDK Data Stream Protocol** 而非自定义 SSE：前端 useChat 开箱即用，省 Day 7 的协议联调；协议简单（行级前缀 + JSON），Python 这边手写 ~30 行就够
- **Anthropic AsyncAnthropic（不是 sync 包 to_thread）**：FastAPI 全异步栈下原生 async client 更顺，避免线程池调度延迟
- **chat orchestrator 不复用 salvaged llm/factory.py**：streaming 接口不在 LLMProvider 抽象里（complete 是 sync 一次返回）；v0.2 给 LLMProvider 加 stream_complete 抽象方法时再 unify
- **ChatStreamRequest 保留 citations 字段**：前端会带回当前 workspace 的引用源（[N] → URL）；后端用它注入 prompt + 在 finish chunk 回传

### 遗留

- 真 `curl localhost:8000/api/v1/chat/stream` 端到端冒烟需要 ANTHROPIC_API_KEY，待用户本地 venv 就绪
- LangGraph 在 v0.1 utilization 偏低（一个 prepare 节点）；v0.2 加 industry/market agents 时收益才显现
- 暂没接 OpenAI / DeepSeek 的 streaming（v0.1 默认 Claude 落地）；v0.2 切其他 provider 时给 LLMProvider 加 stream_complete
- prompt template 里 cards 的 schema 假设较紧（financial_kpi / announcement_timeline / research_report 三种）；新卡片类型加进来要同步改 follow_up.j2

### 决策

- **PRD 实际是 3 张卡（不是计划 §7 写的 5 张）**：5 是早期粗估，PRD §3 + architecture.md §1 都明确 3 张（财务 / 公告 / 研报）；按 PRD 实施。NEXT_STEPS 已订正
- **Pydantic schemas 单独放 routes/_schemas.py**：不直接 reuse interfaces.py 的 dataclass。理由：服务内部用 dataclass（轻量，可静态推导），HTTP 边界用 Pydantic（严格校验 + OpenAPI doc）。两层各自演进，互不污染
- **DeepSeek 不写独立 provider**：直接复用 `OpenAIProvider` + 改 base_url 即可（DeepSeek API 兼容 OpenAI）。少一个 provider 类，少一组测试
- **stock route 不直接 import provider**：通过 service 层间接调用，符合三层 routes → services → repositories 单向依赖
- **CORS_ORIGINS 用逗号分隔字符串而非 list**：Pydantic v2 + .env 文件不支持直接 list 解析，需要自定义 validator；用字符串 + 计算属性更朴素
